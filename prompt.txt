ВАЖНО — ТЕКУЩИЙ TTS‑СТЕК (НИКАКОГО PIPER!)
═══════════════════════════════════════════════════════════════

Мы БОЛЬШЕ НЕ используем локальный Piper TTS. Любой код, упоминания, файлы и зависимости, связанные с Piper, должны игнорироваться или удаляться. Аудио синтезируется ТОЛЬКО через:

1. Казахский язык (kk-KZ) → Yandex SpeechKit TTS (cloud REST API).
   Эндпоинт: https://tts.api.cloud.yandex.net/speech/v1/tts:synthesize
   Auth: Authorization: Api-Key <YANDEX_API_KEY>  (читать из process.env.YANDEX_API_KEY)
   Параметры: lang=kk-KK (или поддерживаемый Yandex), voice=<подходящий>, format=oggopus, sampleRateHertz=48000.

2. Русский язык (ru-RU) → браузерный Web Speech API (window.speechSynthesis), БЕЗ серверных запросов вообще. Используется SpeechSynthesisUtterance с lang='ru-RU' и подходящим русским голосом.

═══════════════════════════════════════════════════════════════
ПРОБЛЕМА
═══════════════════════════════════════════════════════════════

Пользователь жмёт фразу — звук стартует через 5–10 минут (для длинных текстов) или 5–10 секунд (для коротких). Причина:
- Текст целиком уходит в Yandex и сервер ждёт ВЕСЬ ответ перед отдачей клиенту.
- Клиент ждёт ВЕСЬ blob перед audio.play().
- Нет разбиения на предложения, нет стриминга, нет кеша, нет прогрева.

ЦЕЛЬ: time-to-first-audio = 300–800 мс. Полное озвучивание любого реалистичного текста — 3–5 секунд до старта воспроизведения, дальше — без пауз (gapless).

──────────────────── 1. Бекенд (server/index.js) ────────────────────

- Полностью УДАЛИ spawn Python‑процесса, переменные piperProcess, piperReady, PIPER_PORT, PYTHON_EXE, MODEL_PATH, PIPER_SCRIPT, функцию proxyToPiper и всё, что с ними связано.
- Добавь модуль server/yandex-tts.js: функция synthesizeChunk(text, { lang, voice, format }) → возвращает Promise<Buffer> с аудио (oggopus). Использует global fetch (Node 20+). Auth через Api-Key из process.env.YANDEX_API_KEY. Таймаут 15 с. Ретрай 1 раз на 5xx/network ошибки.
- Утилита splitText(text): сначала по [.!?…\n], потом длинные предложения по [,;:] с лимитом 120 символов. Юникод‑safe. Сохраняет порядок. Возвращает массив непустых строк.
- Эндпоинт POST /api/tts/stream:
  * Body: { text, lang, voice? }
  * Если lang начинается с 'ru' → отвечает 400 { error: 'use_browser_tts' } (русский должен играться браузером).
  * Делит текст через splitText.
  * Параллельный синтез: пул из 3 одновременных запросов к Yandex (через простой semaphore написать руками, БЕЗ npm‑зависимости). Результаты собираются в массив futures, отправляются клиенту СТРОГО ПО ПОРЯДКУ: ждём future[i], пишем 4 байта uint32 BE (длина) + байты в res, flush, идём к future[i+1].
  * Заголовки: Content-Type: application/octet-stream, X-Audio-Format: oggopus, X-Sample-Rate: 48000, X-Chunk-Count: <N>. Transfer-Encoding: chunked включится автоматически когда не задан Content-Length.
  * decision: фрейминг 4‑байтовый length-prefix перед каждым чанком — простой парсинг на клиенте, без MSE.
- Эндпоинт POST /api/tts (legacy fallback): один запрос в Yandex со всем текстом, отдаёт целый opus‑буфер. Оставь рабочим.
- LRU‑кеш в памяти (Map с переупорядочиванием), ключ = sha256(`${lang}|${voice}|${text}`), значение = Buffer чанка. Cap: 200 записей, ~100 МБ суммарно. Проверяй кеш ПЕРЕД каждым запросом к Yandex (на уровне отдельного чанка — это даёт максимальный hit-rate, потому что одинаковые предложения встречаются часто).
- Дисковый кеш: ROOT/tts-cache/<sha>.opus — пиши после успешного синтеза, читай при cold start. Атомарная запись через .tmp + fs.rename.
- Эндпоинт POST /api/tts/prewarm: { phrases: string[], lang } — синтезирует все в кеш, не блокируя (return 202 сразу, работа в фоне).
- Эндпоинт GET /api/health → { status: 'ok', yandexConfigured: !!process.env.YANDEX_API_KEY }.
- На старте сервера (после listen) — если YANDEX_API_KEY есть, асинхронно дёрни prewarm для QUICK_PHRASES (импортируй из общего модуля, см. ниже).
- Логи с тегом [yandex] и тайм‑стампами: time-to-first-yandex-byte, total chunk duration. Никаких console.log внутри hot path без меры.
- MAX_TEXT_LENGTH = 5000.

──────────────────── 3. Фронтенд (src/hooks/useSpeech.js) ────────────────────

Полностью перепиши useTextToSpeech. Логика speak(text, lang):

A. Если lang.startsWith('ru'):
   - Использовать window.speechSynthesis. Создать SpeechSynthesisUtterance(text), lang='ru-RU', выбрать русский голос из speechSynthesis.getVoices() (учесть, что voices грузятся асинхронно — слушать voiceschanged один раз и кешировать выбранный голос).
   - speechSynthesis.cancel() перед speak. onstart → setIsPlaying(true), onend → setIsPlaying(false).
   - Никаких сетевых запросов.

B. Иначе (kk-KZ и любой другой) — Yandex стрим:
   - Лениво создаём AudioContext (по первому пользовательскому жесту), храним в useRef. Если ctx.state === 'suspended' → await ctx.resume().
   - fetch('/api/tts/stream', { method:'POST', headers:{'Content-Type':'application/json'}, body: JSON.stringify({text,lang}), signal: abortController.signal }).
   - Читаем response.body.getReader() побайтово, парсим length-prefix фрейминг: накапливаем буфер, читаем 4 байта uint32 BE = длина чанка, дальше столько байт opus‑данных. Повторяем пока поток не закроется.
   - Каждый собранный opus‑чанк декодируем через ctx.decodeAudioData(arrayBuffer.slice()) → AudioBuffer.
   - Поддерживаем nextStartTime: для каждого AudioBuffer создаём BufferSource, source.connect(ctx.destination), startAt = Math.max(ctx.currentTime + 0.02, nextStartTime); source.start(startAt); nextStartTime = startAt + buffer.duration. Gapless playback по мере поступления.
   - На первом успешно сыгранном чанке: setIsLoading(false), setIsPlaying(true).
   - Храним массив активных sources для stop(). Освобождаем ссылки в source.onended.

stop():
   - speechSynthesis.cancel() (на всякий)
   - abortController.abort()
   - Для каждого активного BufferSource: try { src.stop(0); } catch {}
   - Очищаем очередь, сбрасываем nextStartTime = 0.

Состояния: isPlaying, isLoading, isStreaming, error.

useEffect cleanup → stop() + ctx?.close() (только при unmount).
